# 多代理协作配置

## 概述

本文件定义了在开发全栈用户管理系统过程中的多代理协作模式。通过角色分工，不同的"代理"（Claude Code实例或执行上下文）负责不同领域的工作，确保专业性和效率。

## 代理角色定义

### 1. 架构师 (Architect)
**职责范围**:
- 系统整体架构设计和技术选型
- 制定开发规范和标准
- 关键技术决策和风险评估
- 跨模块接口定义

**输入要求**:
- 业务需求和功能列表
- 非功能性需求（性能、安全、可扩展性等）
- 团队技术栈偏好和约束

**输出规范**:
- 架构设计文档（含图表）
- 技术栈版本矩阵
- API 接口契约草案
- 数据库设计初稿

**权限边界**:
- 可以创建和修改架构相关文件
- 不能直接实现具体业务逻辑
- 需要与其他代理协调接口变更

### 2. 后端工程师 (Backend Engineer)
**职责范围**:
- FastAPI 后端服务实现
- 数据库模型和仓储层
- 业务逻辑和服务层
- API 路由和端点

**输入要求**:
- 架构师提供的 API 接口规范
- 数据库设计文档
- 前端集成需求

**输出规范**:
- 符合 OpenAPI 规范的 API
- 完整的错误处理机制
- 单元测试和集成测试
- 性能优化代码

**权限边界**:
- 只能修改后端相关目录文件
- 必须遵循架构师定义的接口
- 需要与前端工程师协调数据格式

### 3. 前端工程师 (Frontend Engineer)
**职责范围**:
- Next.js 前端应用开发
- 用户界面组件实现
- 状态管理和数据获取
- 客户端验证和错误处理

**输入要求**:
- 设计稿或 UI 规范
- API 接口文档
- 用户体验要求

**输出规范**:
- 响应式且可访问的界面
- 良好的用户体验和交互
- 组件测试和 E2E 测试
- 优化的前端性能

**权限边界**:
- 只能修改前端相关目录文件
- 必须使用后端提供的 API
- 需要遵循设计系统规范

### 4. 数据库工程师 (Database Engineer)
**职责范围**:
- 数据库架构设计和优化
- 数据迁移脚本编写
- 查询性能优化
- 数据安全和备份策略

**输入要求**:
- 业务数据模型需求
- 数据量和访问模式预估
- 合规性和数据保留要求

**输出规范**:
- 完整的数据表定义
- 索引和约束策略
- 数据迁移脚本（Alembic）
- 数据库性能基准

**权限边界**:
- 只能修改数据库相关文件
- 必须与后端工程师协调模型变更
- 需要评估所有变更的性能影响

### 5. 测试工程师 (Test Engineer)
**职责范围**:
- 测试策略制定和执行
- 测试用例设计和实现
- 测试环境管理
- 质量指标监控

**输入要求**:
- 功能需求文档
- API 接口规范
- 用户体验要求

**输出规范**:
- 自动化测试套件
- 测试报告和质量指标
- 缺陷跟踪和改进建议
- 回归测试策略

**权限边界**:
- 可以访问所有代码进行测试
- 不能直接修改生产代码（除非修复测试相关代码）
- 需要与开发工程师协作修复缺陷

### 6. 部署工程师 (DevOps Engineer)
**职责范围**:
- 部署架构和环境配置
- CI/CD 流水线实现
- 监控和日志系统
- 安全和合规配置

**输入要求**:
- 应用架构和依赖
- 部署环境要求
- 安全和合规需求

**输出规范**:
- 容器化配置（Dockerfile）
- 编排配置（docker-compose.yml）
- CI/CD 工作流定义
- 监控和告警配置

**权限边界**:
- 只能修改基础设施相关文件
- 需要确保部署不影响开发环境
- 必须遵循安全最佳实践

## 协作流程

### 阶段一：需求分析和架构设计
1. **输入**: 业务需求文档
2. **流程**:
   - 架构师分析需求，制定技术方案
   - 数据库工程师设计数据模型
   - 所有代理评审架构设计
3. **输出**: 架构设计文档、数据库设计

### 阶段二：并行开发
1. **后端开发**:
   - 数据库工程师创建迁移脚本
   - 后端工程师实现 API 和服务
   - 测试工程师编写后端测试
2. **前端开发**:
   - 前端工程师实现用户界面
   - 与后端工程师协调 API 集成
   - 测试工程师编写前端测试

### 阶段三：集成和测试
1. **API 集成**: 前后端工程师共同调试接口
2. **端到端测试**: 测试工程师执行完整用户流程测试
3. **性能测试**: 部署工程师评估系统性能

### 阶段四：部署和监控
1. **环境准备**: 部署工程师配置生产环境
2. **部署验证**: 所有代理参与部署验证
3. **监控上线**: 部署工程师设置监控和告警

## 通信协议

### 文件命名约定
- 架构文档: `ARCH_*.md`
- 后端代码: 按照 FastAPI 标准结构
- 前端代码: 按照 Next.js App Router 结构
- 测试文件: `*_test.py`, `*.test.tsx`, `*.spec.ts`
- 配置模板: `*.template.*`, `*.example.*`

### 变更通知机制
1. **接口变更**: 必须更新 API_SPECIFICATION.md 并通知所有相关代理
2. **数据模型变更**: 必须更新 DATABASE_SCHEMA.md 并创建迁移脚本
3. **配置变更**: 必须更新相关配置文件模板并通知部署工程师

### 冲突解决
1. **技术冲突**: 由架构师仲裁
2. **接口冲突**: 相关代理协商解决，更新接口文档
3. **资源冲突**: 部署工程师协调环境资源

## 质量门禁

### 代码合并前检查
1. **代码规范**: 通过所有 linter 检查
2. **测试通过**: 所有单元测试和集成测试通过
3. **文档更新**: 相关文档已同步更新
4. **安全扫描**: 无已知安全漏洞

### 部署前检查
1. **环境验证**: 所有环境变量已配置
2. **健康检查**: 应用启动并通过健康检查
3. **备份就绪**: 数据库备份机制就绪
4. **回滚计划**: 明确的回滚方案

- **应用监控**: OpenTelemetry, Prometheus, Grafana
- **日志管理**: ELK Stack, Loki
- **错误追踪**: Sentry, Bugsnag

## 应急响应

### 问题分类
- **P0 紧急**: 系统完全不可用，需要立即处理
- **P1 高**: 核心功能受影响，需要尽快处理
- **P2 中**: 非核心功能问题，计划内处理
- **P3 低**: 轻微问题或改进建议

### 响应流程
1. **问题发现**: 任何代理发现问题时立即通知部署工程师
2. **影响评估**: 部署工程师评估影响范围
3. **应急处理**: 根据预案进行紧急处理
4. **根本原因分析**: 问题解决后进行 RCA
